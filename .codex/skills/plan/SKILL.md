# Plan Skill

현재 워크트리의 `.task.md`를 읽어 구체적인 구현 계획을 수립하고, 코드베이스 탐색을 통해 현실적인 실행 방안을 제시합니다.

## 실행 흐름

### Phase 1: Task 파일 읽기

1. 현재 디렉토리의 `.task.md`를 읽습니다.
   - 없으면 상위 디렉토리에서 탐색합니다 (최대 3단계).
   - 여전히 없으면 사용자에게 알리고 `/start-task`로 먼저 워크트리를 만들도록 안내합니다.
2. 다음 정보를 파악합니다.
   - 작업 유형 (`feat` / `fix` / `hotfix` / `chore` / `refactor`)
   - 대상 앱 (`web` / `admin` / `make` / `sejong`)
   - 구현해야 할 기능 또는 수정해야 할 버그의 핵심 요건

### Phase 2: 코드베이스 탐색

스펙에서 파악한 범위에 따라 관련 코드를 탐색합니다.

**탐색 우선순위**:
1. **라우트** (`src/routes/`) – 관련 페이지/레이아웃 파악
2. **컴포넌트** (`src/components/`, `-components/`) – 재사용 가능한 컴포넌트 파악
3. **훅** (`src/hooks/`) – 기존 상태·비즈니스 로직 파악
4. **API / 스키마** (`src/api/`, `src/schema/`) – 엔드포인트·타입 파악
5. **공용 UI** (`packages/ui`) – 사용 가능한 컴포넌트 파악
6. **i18n** (`src/locales/`) – 기존 번역 키 파악 (UI 텍스트가 있는 경우)

**탐색 시 확인 사항**:
- 동일하거나 유사한 기존 구현이 있는지
- 따라야 할 패턴(라우트 구조, API 호출 방식, 스타일링 패턴)
- 수정할 때 함께 변경해야 하는 연관 파일

### Phase 3: 명확화 질문 (필요 시)

다음 상황에서 사용자에게 질문합니다.

- **스펙이 구체적이지 않을 때**: 예) UI 동작, 에러 처리 방식, 엣지 케이스
- **구현 방식이 여러 가지일 때**: 예) 새 컴포넌트 vs 기존 컴포넌트 확장, 클라이언트 상태 vs URL 상태
- **백엔드 API가 불명확할 때**: 예) 아직 없는 엔드포인트, 응답 스키마 미확정
- **영향 범위가 넓을 때**: 예) 공용 컴포넌트 수정 시 다른 화면에 미치는 영향

질문은 한 번에 묶어서 하고, 진행을 막는 것만 선별해서 묻습니다.

### Phase 4: 계획 수립

탐색 결과를 바탕으로 구체적인 계획을 작성합니다.

**계획 구성 요소**:

```markdown
## 구현 계획: [작업 제목]

### 요약
[1~3줄로 무엇을, 왜 만드는지]

### 영향 범위
- 앱: web | admin | make | sejong
- 신규 파일: N개
- 수정 파일: N개

### 신규 파일
| 파일 경로 | 역할 |
|-----------|------|
| src/routes/.../new-page.tsx | 새 페이지 라우트 |
| src/api/authenticated/feature.ts | API 함수 |
| src/schema/api/feature/feature.ts | Zod 스키마 |

### 수정 파일
| 파일 경로 | 변경 내용 |
|-----------|-----------|
| src/routes/.../route.tsx | 네비게이션 링크 추가 |
| src/locales/ko/common.json | 번역 키 추가 |

### 구현 순서
1. Zod 스키마 정의 (`src/schema/api/...`)
2. API 함수 작성 (`src/api/...`)
3. 라우트 파일 생성 (`src/routes/...`)
4. 컴포넌트 구현
5. i18n 키 추가 (필요 시)
6. 타입 체크 / 린트 검증

### 주요 고려사항
- [패턴, 엣지 케이스, 주의할 점]

### 미확정 사항
- [백엔드 API 스펙 확정 필요 등 외부 의존성]
```

### Phase 5: 사용자 승인 및 저장

1. 계획을 사용자에게 보여주고 승인을 받습니다.
   - 승인이면 `.plan.md`에 저장하고 구현을 시작합니다.
   - 수정 요청이 있으면 해당 부분을 조정 후 재확인합니다.
2. `.plan.md`에 계획을 저장합니다.
   - 경로: 현재 워크트리 루트 (`.task.md`와 같은 위치)
   - 계획 전문을 그대로 저장합니다.
3. 구현 시작 안내를 출력합니다.

```text
계획이 저장되었습니다: .plan.md

구현을 시작하려면:
  /implement 또는 직접 파일 작업을 시작하세요.

막히는 부분이 있으면 언제든지 물어보세요.
```

## 주의사항

- **탐색 먼저, 추측 금지**: 파일을 읽지 않은 채 패턴을 가정하지 않습니다.
- **기존 패턴 우선**: 새 패턴 도입보다 프로젝트 기존 패턴을 따릅니다.
- **질문은 한 번에**: 여러 번 나눠 묻지 않고 필요한 질문을 한 번에 묶어 합니다.
- **계획은 실행 가능해야 함**: 파일 경로·역할이 구체적이어야 합니다. "적절히 수정" 같은 모호한 표현은 사용하지 않습니다.
- **미확정 사항 명시**: 백엔드 API 미확정 등 외부 의존성은 계획에 명시하고, 모킹/임시 처리 방안을 함께 제시합니다.
- **`.task.md` 없을 때**: 구현을 강행하지 않고 `/start-task` 먼저 실행하도록 안내합니다.
- **공용 패키지 영향 확인**: `packages/ui` 컴포넌트를 수정할 때는 모든 앱의 영향 범위를 명시합니다.
