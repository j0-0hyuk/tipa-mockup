# Submit PR Skill

프론트엔드 변경사항을 검증하고(`pnpm`/`turbo` 기반), 꼭 필요한 수정만 반영한 뒤 PR을 생성합니다.

## 실행 흐름

### Phase 1: 사전 검증

1. **변경사항 수집** (중복 제거 포함)
   ```bash
   changed_files=$(
     (
       git diff --name-only                     # 스테이징되지 않은 변경
       git diff --cached --name-only           # 스테이징된 변경
       git ls-files --others --exclude-standard  # 새 파일
     ) | sort -u
   )
   ```
2. **변경 파일이 없으면** 즉시 중단하고 사용자에게 알림
3. **영향 워크스페이스 식별**: `apps/*`, `packages/*` 기준으로 영향 범위를 추출
4. **현재 브랜치 확인**: 이미 피처 브랜치에 있으면 Phase 4에서 새 브랜치 생성을 건너뜀

### Phase 2: 정적 검증 (lint/type/build)

변경된 워크스페이스에 맞춰 필요한 검증만 실행합니다.

1. **앱/패키지별 검증 명령**
   - `apps/web`
     - `pnpm --filter @docs-front/web lint`
     - `pnpm --filter @docs-front/web check-types`
     - `pnpm --filter @docs-front/web build`
   - `apps/admin`
     - `pnpm --filter admin lint`
     - `pnpm --filter admin build`
   - `apps/make`
     - `pnpm --filter make lint`
     - `pnpm --filter make check-types`
     - `pnpm --filter make build`
   - `apps/sejong`
     - `pnpm --filter sejong lint`
     - `pnpm --filter sejong build`
   - `packages/ui`
     - `pnpm --filter @docs-front/ui lint`
     - `pnpm --filter @docs-front/ui check-types`
     - `pnpm --filter @docs-front/ui build`
     - 필요 시 `pnpm test:projects`
2. **루트/공용 설정 변경 시 추가 검증**
   - `pnpm lint`
   - `pnpm build`
3. 에러가 있으면 **자동 수정 가능한 범위만 수정** 후 재검증
4. 자동 수정 불가능한 에러는 사용자에게 보고하고 중단

### Phase 3: 코드 리뷰 (최대 2회 반복)

각 반복(iteration)에서:

1. **변경된 파일에 대해 `code-review` 스킬을 직접 실행**
   - `code-review` 스킬의 **최신 리뷰 프로세스 전체**를 그대로 따른다.
   - `submit-pr`에서 리뷰 단계(패스), 하위 스킬 호출 순서, 체크리스트를 재정의/복제하지 않는다. (Single Source of Truth: `code-review`)
   - `code-review` 결과에 포함된 검증 명령 제안은 Phase 2에서 이미 수행한 검증과 중복 여부를 확인해, 누락된 검증만 추가 실행한다.

2. **결과 평가 및 처리**
   - 🔴 **Critical 발견** → 자동 수정 (기계적으로 안전한 항목만)
   - 🟡 **Warning 발견** → **사용자에게 목록을 보여주고 수정 여부를 확인받음**
     - 각 Warning 항목별로 문제 설명과 수정 제안을 제시
     - 사용자가 선택한 항목만 수정 진행
     - 수정하지 않기로 한 항목은 PR 본문에 "Known Issues"로 기록
   - 🔵 **Info만 존재** → 수정하지 않음, Phase 4로 진행
   - ✅ **이슈 없음** → Phase 4로 진행

3. **반복 제한**: 2회 iteration 후에도 Critical 이슈가 남아있으면
   - 남은 이슈를 사용자에게 보고
   - 사용자 확인 후 진행 여부 결정

### Phase 4: PR 생성

1. **브랜치 처리**
   - 이미 피처 브랜치에 있으면 해당 브랜치 사용
   - `main`에 있으면 변경 내용에 기반해 브랜치 생성
   - 형식: `feat/기능명`, `fix/수정내용`, `chore/작업내용` 등

2. **커밋 & 푸시**

   ```bash
   # 피처 브랜치가 아닌 경우에만
   git checkout -b <branch-name>

   # 변경된 파일만 명시적으로 추가 (git add -A 사용 금지)
   git add <file1> <file2> ...
   git commit -m "<한글 conventional commit message>"
   git push origin <branch-name>
   ```

   - `git log`에서 최근 커밋 패턴(type/scope/형식)을 확인하고 동일 컨벤션을 준수
   - 커밋 메시지는 반드시 한글로 작성 (예: `fix(auth): 리프레시 토큰 만료 검증 오류 수정`)

3. **PR 생성**

   ```bash
   gh pr create --title "<PR 제목>" --body "<PR 본문>" --base main
   ```

4. **PR 본문 작성 전 필수: 코드베이스 확인**

   PR 본문은 **반드시 실제 코드를 읽고 작성**합니다. 커밋 메시지나 추측으로 작성하지 않습니다.
   - `git diff origin/main...HEAD`로 전체 변경 내용 확인 (merge-base 기준)
   - 라우트 파일에서 실제 네비게이션 구조 확인 (탭인지 사이드바인지, 어떤 경로인지)
   - 컴포넌트에서 실제 UI 텍스트 확인 (하드코딩 문자열, i18n 키의 실제 값)
   - API 호출 흐름, 상태 변화, 조건 분기 등 동작 로직 확인

5. **PR 본문에 포함할 내용**

   ```markdown
   ## 변경 사항

   [사용자/QA 관점에서 "화면에서 무엇이 어떻게 바뀌는지" 서술]
   [기술 구현이 아닌 동작 결과 중심으로 작성]

   - **굵은 글씨로 핵심 변경** 강조 후 부연 설명
   - 네비게이션 변경은 실제 위치를 정확히 명시 (예: "사이드바에 메뉴 추가" vs "탭 바에 탭 추가")
   - UI 텍스트 변경은 실제 문구를 인용

   ---

   ## QA 안내

   | 항목             | 내용                                                             |
   | ---------------- | ---------------------------------------------------------------- |
   | 테스트 환경      | <https://bizcanvas.net>                                          |
   | 배포 브랜치      | `<현재 브랜치명>`                                                |
   | 변경 기능 진입점 | [실제 경로와 UI 위치. 예: `/credit-plan` 탭 바 > `친구 초대` 탭] |

   ### 사전 준비 (필요한 경우에만 작성)

   [테스트에 특별한 조건이 필요한 경우만 작성, 필요 없으면 이 섹션(헤더 포함) 삭제]

   - 필요한 계정 수, 역할, 권한
   - 테스트 데이터 준비 방법
   - 테스트 흐름 요약 (예: A가 링크 공유 → B가 접속 → B가 결제 → A에 반영)

   ### ✅ 기능 테스트 (주요 변경사항)

   **1. [기능 영역 이름]**

   - [ ] [구체적 조작] → [기대 결과]

   **2. [기능 영역 이름]**

   - [ ] [구체적 조작] → [기대 결과]

   ### ⚠️ 예외/엣지 케이스

   - [ ] [비정상 입력/상황] → [기대 동작]

   ### 🔁 회귀 테스트 (기존 기능 영향도)

   - [ ] [기존 기능]이 기존과 동일하게 정상 동작하는지 확인

   ---

   ## 코드 리뷰 결과

   - ✅ 빌드 통과
   - ✅ 테스트 통과
   - ✅ 코드 리뷰 통과 (N회 iteration)
   - ⚠️ Known Issues (사용자가 스킵한 Warning 항목, 있는 경우)

   ## 변경된 파일

   - `파일1` - 설명
   - `파일2` - 설명
   ```

6. **QA 체크리스트 작성 원칙**
   - **코드 기반 작성**: 실제 라우트 파일, 컴포넌트, i18n JSON을 읽고 정확한 경로/문구/동작을 기재
   - **기능별 번호 그룹핑**: 관련 체크 항목을 `**1. 기능명**` 형태로 묶어서 구조화
   - **비개발자 관점**: "무엇을 해서 어떤 결과가 나와야 하는지"로 서술, 코드 용어(state, API, localStorage 등) 최소화
   - **사전 준비 분리**: 여러 계정/특정 데이터 등 테스트 전제 조건이 있으면 `### 사전 준비` 섹션에 별도 기재
   - **진입점 정확히**: 라우트 파일에서 실제 탭/메뉴/경로를 확인하여 기재 (짐작하지 않는다)

## 자동 수정 범위

### 자동 수정 O (사용자 확인 없이 진행)

- 린트/타입 오류 중 기계적으로 안전한 수정
- 명확한 런타임 에러 방지(null 체크, import 경로 오류)
- 민감 정보 하드코딩 제거
- 단순 중복/오탈자/불필요 코드 제거

### 사용자 확인 필요 (Warning → 선택적 수정)

- Query key/캐시 전략 변경
- 성능 최적화로 동작 타이밍이 바뀌는 수정
- UI 마이그레이션(`@docs-front/ui` -> `@bichon/ds`) 제안
- 라우팅/권한 플로우 변경
- UX/문구/동선 변경

### 자동 수정 X (보고만)

- 결제/크레딧/요금제 핵심 비즈니스 로직 변경
- 인증/권한 정책 변경
- 대규모 아키텍처 변경
- PR 범위를 크게 벗어나는 리팩토링

## 주의사항

- **git add -A 금지**: 변경된 파일만 명시적으로 staging (`logs/`, `tools/` 등 무관한 파일 포함 방지)
- **base 브랜치**: 항상 `main` 대상으로 PR 생성
- **커밋 히스토리**: 리뷰 수정사항은 필요 시 `fix:` 커밋으로 남기고 PR에서 squash 가능하도록 유지
- **안전장치**: PR 생성 전 항상 사용자에게 최종 확인 요청
- **과도한 수정 금지**: 수정할 가치가 있는 것만 수정. 동작에 문제가 없는 스타일 이슈는 무시
- **리스크 우선순위**: 인증/권한, 결제/크레딧, 데이터 노출, 런타임 장애 리스크를 최우선으로 점검
