# TIPA 신청자격 점검 플로우 기획

## 목적

- 공고 추천 이후 `지원 가능성`을 빠르게 판단한다.
- 반복되는 자격 요건은 `/company`의 섹션 4에서 한 번만 받는다.
- 공고마다 다른 예외 조항과 분기 조건은 `초안 작성하기` 직전 모달에서 해소한다.
- 더미데이터 단계에서도 "추천"과 "실제 신청 가능성"이 분리되어 보이게 한다.

## 문서 근거

바탕화면 `기술개발사업신청자격` 폴더의 HWPX 11개를 확인했다. 중복이 많고, 실제 반복 축은 아래로 압축된다.

### 반복되는 공통 축

- 주관/공동 연구개발기관 신청자격
- 중소기업 여부, 사업자등록증 기준 신청
- 참여제한 여부
- 기술료, 환수금, 제재부가금, 성과입력 등 의무이행 여부
- 휴업, 폐업, 부도, 체납, 채무불이행 여부
- 부채비율 1000% 이상, 자본전액잠식 여부
- 기개발, 기지원, 중복신청 여부
- IRIS, NRI, 국가연구자번호, 기관등록 준비 여부

### 공고별로 갈리는 분기 축

- 업력 7년 초과 10년 이하 기업의 `신산업 창업분야` 적합 여부
- `R&D 졸업제` 및 최근 7년 수행 이력
- 재무 예외 사유
  - 창업 3년 미만
  - 디딤돌 첫걸음
  - 채권은행협약
  - 이행보증보험증권
  - 산업위기지역
  - 투자, 시설확장 등에 따른 일시적 재무악화

## 역할 분리 원칙

- 섹션 4는 `기본 자격 진단`이다.
- 모달은 `선택 공고의 추가 확인`이다.
- 같은 질문을 두 번 묻지 않는다.
- 공고별 모달은 1개 공고당 최대 3문항으로 제한한다.

## 화면 구조

## 1. 섹션 4: 신청자격 사전 점검

4개의 카드형 블록으로 나눈다.

### 4-1. 신청 주체와 기관 구성

- 질문 예시
  - 중소기업확인서 기준으로 신청 가능한 기업인가요?
  - 주관/공동 기관 구성에 비영리기관이 들어가나요?
  - 사업자등록증 기준 신청 주체가 맞나요?
- 성격
  - blocker

### 4-2. 참여제한과 의무이행

- 질문 예시
  - 대표자, 연구책임자, 참여기관 중 현재 참여제한 중인 사람이 있나요?
  - 기술료, 환수금, 제재부가금, 성과입력 미이행 건이 남아 있나요?
  - 국세, 지방세, 4대보험, 임금체불 등 체납 이슈가 있나요?
- 성격
  - blocker 중심
  - 일부는 평가 개시 전 해소 가능하므로 warning도 포함

### 4-3. 재무와 경영 상태

- 질문 예시
  - 최근 결산 기준 부채비율 1000% 이상인가요?
  - 자본전액잠식 상태인가요?
  - 휴업, 폐업, 부도, 회생절차 신청 상태인가요?
- 성격
  - warning 우선
  - 예외 조항이 많아 모달 분기로 넘길 수 있게 설계

### 4-4. 과제 중복과 수행 이력

- 질문 예시
  - 동일하거나 유사한 과제를 이미 수행했나요?
  - 동일 모집차수에 중복 신청 중인가요?
  - 최근 수혜 이력 때문에 졸업제에 걸릴 가능성이 있나요?
- 성격
  - blocker + warning 혼합

## 2. 페이지 상단 준비 체크

IRIS/NRI 관련 항목은 탈락 판정보다 준비도에 가깝다. 섹션 4 본문 질문 대신 별도 준비 체크로 배치한다.

- IRIS 회원가입
- 연구자 전환 동의
- 국가연구자번호 발급
- 기관등록, 기관대표자, 총괄담당자 등록

UI는 작은 체크리스트나 배너로 처리한다.

## 상태 모델

추천 점수와 별개로 신청 상태를 따로 둔다.

- `eligible`
  - 바로 초안 작성 가능
- `needs_check`
  - 공고별 추가 확인 필요
- `needs_action`
  - 보완 후 진행 권장
- `ineligible`
  - 현재 조건으로 신청 어려움

카드에는 점수 옆에 이 상태를 배지로 보여준다.

## 모달 규칙

모달은 `needs_check` 또는 `needs_action`일 때만 띄운다.

### 모달 제목

- `이 공고는 추가 확인이 필요해요`

### 모달 설명

- `기본 신청 자격은 먼저 확인했어요. 이 공고에만 적용되는 조건 2가지를 더 확인할게요.`

### 버튼

- 기본: `확인하고 초안 작성하기`
- 보조: `다른 공고 보기`

## 모달 질문 카탈로그

실제 HWPX 근거 기준으로는 아래 id가 우선순위다.

### `new-industry-fit-7to10`

- 대상
  - 업력 7년 초과 10년 이하 기업에만 노출
- 질문
  - `귀사는 이 공고가 요구하는 신산업 창업분야에 해당하나요?`
- 결과
  - 아니오면 `ineligible`

### `graduation-limit`

- 대상
  - 최근 정부 R&D 수혜 이력이 있는 기업
- 질문
  - `최근 7년 수행 이력 기준으로 R&D 졸업제 한도에 걸리지 않나요?`
- 결과
  - 아니오면 `needs_check` 또는 `ineligible`

### `finance-exception`

- 대상
  - 부채비율, 자본잠식, 재무 이슈 응답이 있었던 기업
- 질문
  - `아래 예외 사유 중 해당되는 항목이 있나요?`
- 선택지
  - 창업 3년 미만
  - 디딤돌 첫걸음
  - 채권은행협약
  - 이행보증보험증권 제출 가능
  - 산업위기지역
  - 일시적 재무악화
- 결과
  - 예외 있으면 `needs_action`
  - 없으면 `ineligible`

### `single-call-limit`

- 대상
  - 동일 사업, 동일 모집차수 중복 신청 가능성이 있는 경우
- 질문
  - `이번 모집차수에 주관기관으로 다른 과제를 이미 신청했나요?`
- 결과
  - 예면 `ineligible`

## 더미데이터 구조 제안

```ts
type EligibilityStatus = 'eligible' | 'needs_check' | 'needs_action' | 'ineligible';

interface CommonEligibilityAnswer {
  id: string;
  answer: 'yes' | 'no' | 'unknown';
}

interface ProgramClause {
  id:
    | 'new-industry-fit-7to10'
    | 'graduation-limit'
    | 'finance-exception'
    | 'single-call-limit';
  severity: 'blocker' | 'warning';
}

interface ProgramUiModel {
  id: string;
  title: string;
  score: number;
  eligibilityStatus: EligibilityStatus;
  specialClauses: ProgramClause[];
}
```

## 추천 UI 흐름

1. 기업정보 입력
2. 섹션 4에서 공통 자격 진단
3. 추천 공고 카드에 점수와 신청 상태 표시
4. `초안 작성하기` 클릭
5. 공고별 모달에서 특수 조항 확인
6. 통과 시 `/start2`로 이동

## 구현 우선순위

1. 현재 섹션 4 질문을 이 4블록 구조에 맞춰 라벨과 의미를 정리
2. 공고 모달 조항을 실제 HWPX 근거 id 위주로 정리
3. 카드 상태 배지와 요약문구 추가
4. `needs_check`와 `needs_action` 분기 노출

## 메모

- `tech-fee`, `security-grade`, `youth-hire` 같은 조항은 현재 바탕화면 HWPX 묶음의 공통 반복축이라기보다 별도 공고나 타 자료 기반일 가능성이 높다.
- 이번 설계의 기준 문서는 바탕화면 HWPX 묶음이므로, 공고별 모달 질문도 우선 이 기준에 맞춰 단순화하는 게 맞다.
